## 1. Executive Summary

### 1.1 제품 비전

건강상담도우미는 AI 에이전트 기반의 건강상담 서비스로, "증상 인지부터 병원 방문까지"의 전 과정을 하나의 대화형 경험으로 연결합니다. 사용자는 복잡한 의료 정보 탐색 없이 AI와의 대화만으로 적절한 진료과를 안내받고, 주변 병의원을 찾아 방문할 수 있습니다.

### 1.2 문제 정의

| 문제 | 현재 상황 | 영향 |
|------|----------|------|
| 진료과 선택 어려움 | 일반인은 증상만으로 어떤 진료과를 방문해야 하는지 판단하기 어려움 | 잘못된 진료과 방문 → 재방문, 시간·비용 낭비 |
| 병원 정보 파편화 | 병원 검색, 위치 확인, 진료시간 조회가 각각 다른 플랫폼에 분산 | 정보 수집에 과도한 시간 소요 |
| 사전 문진 부재 | 병원 방문 전 자신의 증상을 체계적으로 정리할 수단 부족 | 진료 시 증상 전달 누락, 진료 효율 저하 |
| 의료 접근성 격차 | 고령자, 의료 취약 지역 거주자는 적절한 병원을 찾기 더욱 어려움 | 적시 진료 기회 상실 |

### 1.3 솔루션 개요

AI 멀티에이전트 시스템을 활용하여 다음을 자동화합니다:

1. **표준 문진표 기반 AI 건강상담** — 사용자 증상을 체계적으로 수집
2. **증상-진료과 자동 매핑** — 문진 결과를 분석하여 적합한 진료과 추천
3. **위치 기반 병의원 안내** — 지역 입력 시 주변 병의원 3개 추천
4. **병원 상세 정보 원스톱 제공** — 기본 정보, 지도, 웹페이지를 한 화면에 통합

### 1.4 성공 지표 (KPI)

| 지표 | 목표 | 측정 방법 |
|------|------|----------|
| 상담 완료율 | ≥ 70% | 문진 시작 대비 병원 상세 정보 조회까지 도달한 비율 |
| 평균 상담 소요 시간 | ≤ 5분 | 서비스 진입 ~ 병원 선택까지 평균 시간 |
| 진료과 추천 정확도 | ≥ 85% | 사용자 피드백 및 전문가 검증 기반 |
| 사용자 만족도 (CSAT) | ≥ 4.0 / 5.0 | 상담 완료 후 만족도 설문 |
| 월간 활성 사용자 (MAU) | 런칭 3개월 후 10,000+ | 서비스 분석 도구 측정 |
| 병원 방문 전환율 | ≥ 30% | 병원 전화/길찾기/예약 클릭 비율 |

## 2. 사용자 분석

### 2.1 타겟 사용자 페르소나

#### 페르소나 1 — "바쁜 직장인" 김민수 (32세)

- **상황**: 며칠째 두통과 어지러움이 있지만 어디로 가야 할지 모름
- **니즈**: 빠르고 간편하게 어떤 과를 가야 하는지 알고 싶음
- **행동 패턴**: 모바일 중심, 점심시간이나 퇴근 후 짧은 시간에 해결하고 싶음
- **핵심 가치**: 속도, 정확성, 편의성

#### 페르소나 2 — "건강 걱정 많은 부모" 이영희 (58세)

- **상황**: 무릎 통증이 심해졌는데 정형외과인지 류마티스인지 모름
- **니즈**: 친절하고 이해하기 쉬운 안내, 가까운 좋은 병원 추천
- **행동 패턴**: 스마트폰 사용은 가능하나 복잡한 UI에 익숙하지 않음
- **핵심 가치**: 신뢰성, 접근성, 친절함

#### 페르소나 3 — "타지 방문자" 박준혁 (27세)

- **상황**: 출장 중 갑자기 피부 발진 발생, 낯선 도시에서 병원을 찾아야 함
- **니즈**: 현재 위치 기반으로 즉시 갈 수 있는 피부과 찾기
- **행동 패턴**: GPS 기반 검색 선호, 즉각적인 결과 기대
- **핵심 가치**: 위치 정확성, 실시간 정보, 즉시성

### 2.2 사용자 시나리오

**주요 시나리오 (Happy Path)**:

```
김민수(32세)는 3일째 두통이 지속되어 건강상담도우미에 접속한다.
→ AI가 두통의 위치, 강도, 동반 증상 등을 문진한다. (약 2분)
→ "신경과 방문을 권장합니다"라는 결과와 함께 근거가 제시된다.
→ "강남역 근처"를 입력하자 주변 신경과 3곳이 표시된다.
→ 가장 가까운 병원을 선택하니 진료시간, 지도, 전화번호가 나온다.
→ 전화 버튼을 눌러 예약 후 퇴근길에 방문한다.
전체 소요 시간: 약 4분
```

**예외 시나리오**:

| 시나리오 | 처리 방식 |
|---------|----------|
| 응급 증상 감지 (흉통, 호흡곤란 등) | 즉시 문진 중단, 119 / 응급실 안내 |
| 증상이 모호하여 진료과 특정 불가 | 내과(일반) 우선 추천 + "증상이 지속되면 재상담" 안내 |
| 검색 지역에 해당 진료과 없음 | 검색 반경 확대 또는 인접 지역 병원 안내 |
| 사용자가 추천 결과에 동의하지 않음 | 재문진 또는 다른 진료과 옵션 제시 |
| 야간/휴일 이용 | 야간 진료 가능 병원 또는 응급의료기관 안내 |

### 3.2 FR-001: 서비스 진입 및 이용 동의

**설명**: 사용자가 서비스에 최초 접속 시 서비스 소개, 이용약관 동의, 의료 면책 고지를 진행합니다.

**수용 조건 (Acceptance Criteria)**:

- AC-1: 서비스 소개 화면에서 건강상담도우미의 목적과 이용 방법이 표시된다.
- AC-2: 필수 동의 항목(이용약관, 개인정보) 미동의 시 서비스 이용이 불가하다.
- AC-3: "본 서비스는 의료 행위가 아닙니다" 면책 고지가 명시적으로 노출된다.
- AC-4: 동의 완료 후 AI 문진 상담 화면으로 자동 전환된다.

**UI/UX 요구사항**:

- 동의 항목은 체크박스 형태, 전체 동의 기능 포함
- 면책 고지는 별도 강조 영역(배너 또는 모달)으로 표시
- 재방문 사용자는 동의 절차 생략 (쿠키/로컬 기반)

### 3.3 FR-002: 표준 문진표 기반 AI 건강상담

**설명**: AI 챗봇이 표준 문진표를 기반으로 사용자의 증상을 체계적으로 수집합니다.

**수용 조건 (Acceptance Criteria)**:

- AC-1: 문진은 자연어 대화형과 가이드형 선택지를 병행한다.
- AC-2: 국가건강검진 문진표 등 표준화된 문진 항목을 활용한다.
- AC-3: 최소 수집 항목: 증상 부위, 증상 설명, 기간, 강도(1~10)
- AC-4: 선택 수집 항목: 동반 증상, 과거 병력, 복용 약물, 연령대, 성별
- AC-5: 문진 진행률을 프로그레스바로 표시한다.
- AC-6: 문진 완료까지 평균 소요 시간이 3분 이내여야 한다.
- AC-7: 문진 중 응급 증상 키워드 감지 시 FR-008로 즉시 전환한다.

**문진 흐름**:

```
Phase 1: 초기 증상 파악
  "어디가 불편하신가요?" → 자유 텍스트 입력
  → AI가 증상 분류 및 후속 질문 생성

Phase 2: 상세 문진 (표준 문진표 기반)
  → 증상 부위 확인
  → 증상 기간 확인 (선택형: 오늘/1~3일/1주/1개월+)
  → 증상 강도 (슬라이더: 1~10)
  → 동반 증상 확인 (다중 선택형)
  → 과거 병력/복용 약물 (선택적)

Phase 3: 확인 및 보완
  → 수집 정보 요약 제시
  → 사용자 확인 / 수정
  → 문진 데이터 확정
```

**참고 문진표 항목 (국가건강검진 기반)**:

- 과거 질환 이력 (고혈압, 당뇨, 심장질환 등)
- 가족력 (부모/형제 주요 질환)
- 흡연/음주 여부
- 운동 습관
- 최근 체중 변화
- 현재 복용 약물

### 3.4 FR-003: 증상-진료과 매핑 및 추천

**설명**: 문진 결과를 분석하여 사용자가 방문해야 할 진료과를 추천합니다.

**수용 조건 (Acceptance Criteria)**:

- AC-1: 1순위 진료과를 반드시 1개 추천한다.
- AC-2: 관련 가능성이 있는 경우 2순위 진료과를 함께 제시한다.
- AC-3: 각 추천에 대한 근거(어떤 증상이 해당 추천의 이유인지)를 명시한다.
- AC-4: 긴급도를 3단계로 표시한다: 일반 / 조기 방문 권장 / 응급
- AC-5: 의료 면책 문구가 추천 결과 하단에 항상 노출된다.
- AC-6: 사용자가 추천에 동의하지 않을 경우 재문진 또는 대안 진료과를 제시할 수 있다.

**진료과 매핑 테이블 (핵심)**:

| 증상 카테고리 | 키워드 예시 | 1차 추천 | 2차 추천 |
|-------------|-----------|---------|---------|
| 호흡기/전신 | 감기, 발열, 기침, 몸살 | 내과 | - |
| 소화기 | 복통, 소화불량, 설사, 구토 | 내과(소화기) | 외과 |
| 심혈관 | 가슴 통증, 두근거림 | 내과(순환기) | 응급실 |
| 시력/안구 | 시력 저하, 충혈, 안구 통증 | 안과 | - |
| 근골격 | 관절통, 요통, 골절 의심 | 정형외과 | 재활의학과 |
| 피부 | 발진, 가려움, 여드름, 탈모 | 피부과 | 알레르기내과 |
| 구강/치아 | 치통, 잇몸 출혈 | 치과 | - |
| 이비인후 | 인후통, 귀 통증, 코막힘 | 이비인후과 | 내과 |
| 두통/신경 | 두통, 어지러움, 저림 | 신경과 | 내과 |
| 비뇨기 | 배뇨 이상, 혈뇨 | 비뇨의학과 | 내과 |
| 정신건강 | 불면, 우울, 불안 | 정신건강의학과 | 내과 |
| 여성 건강 | 생리 불순, 하복부 통증 | 산부인과 | 내과 |
| 소아 | 소아 발열, 발진, 성장 문제 | 소아청소년과 | 해당 전문과 |

**추천 결과 출력 포맷**:

    },
    "secondary": {

    "disclaimer": "본 추천은 AI 기반 참고 정보이며, 정확한 진단은 의료기관에서 받으시기 바랍니다."

### 3.5 FR-004: 위치 기반 병의원 검색

**설명**: 사용자 위치를 기반으로 추천 진료과에 해당하는 주변 병의원 3개를 검색하여 안내합니다.

**수용 조건 (Acceptance Criteria)**:

- AC-1: GPS 자동 감지 또는 주소/지역명 직접 입력을 지원한다.
- AC-2: 추천 진료과에 해당하는 병의원을 최대 3개 표시한다.
- AC-3: 검색 결과는 거리순 정렬을 기본으로 하되, 평점을 함께 고려한다.
- AC-4: 각 병의원에 대해 병원명, 진료과, 거리, 평점, 운영 상태를 표시한다.
- AC-5: 검색 반경 내 해당 진료과 병원이 3개 미만이면 반경을 자동 확대한다.
- AC-6: 야간/휴일에는 현재 진료 가능한 병의원을 우선 표시한다.

**검색 로직**:

```
1. 사용자 위치 확보 (GPS 또는 주소 → 좌표 변환)
2. 반경 설정 (기본 1km, 결과 부족 시 3km → 5km 자동 확대)
3. 추천 진료과 키워드로 병의원 검색
4. 정렬 기준: 거리(50%) + 평점(30%) + 운영 상태(20%)
5. 상위 3개 선별 및 표시
```

**데이터 소스**:

- 건강보험심사평가원 병의원 데이터 (공공데이터포털)
- 네이버/카카오 지도 API (위치, 리뷰, 평점)
- 실시간 운영 정보 (가용 시)

---

### 3.6 FR-005: 병원 상세 정보 표시

**설명**: 사용자가 병원을 선택하면 해당 병원의 상세 정보를 통합 화면으로 제공합니다.

**수용 조건 (Acceptance Criteria)**:

- AC-1: 기본 정보(병원명, 주소, 전화번호, 진료시간, 휴진일)를 표시한다.
- AC-2: 전화번호 터치 시 전화 앱이 실행된다 (모바일).
- AC-3: 네이버 지도 임베드로 병원 위치를 지도에 표시한다.
- AC-4: 현재 위치에서 병원까지 길찾기(도보/대중교통/자가용)를 제공한다.
- AC-5: 병원 공식 웹사이트 링크를 제공한다 (존재 시).
- AC-6: 상담 결과 요약(문진 내용 + 추천 근거)을 함께 표시한다.

---

### 3.7 FR-006: 네이버 지도 연동

**설명**: 병원 위치 표시 및 길찾기를 네이버 지도 API를 통해 제공합니다.

**수용 조건 (Acceptance Criteria)**:

- AC-1: 병원 상세 화면에 네이버 지도가 임베드되어 병원 마커가 표시된다.
- AC-2: 길찾기 버튼 클릭 시 네이버 지도 앱(모바일) 또는 웹 길찾기가 실행된다.
- AC-3: 도보, 대중교통, 자가용 3가지 이동 수단 옵션을 제공한다.
- AC-4: 예상 이동 시간 및 거리를 표시한다.

**필요 API**:

- Naver Maps JavaScript API v3 (지도 임베드)
- Naver Maps Geocoding API (주소 → 좌표)
- Naver Maps Directions API (길찾기)

---

### 3.8 FR-008: 응급 상황 감지 및 안내

**설명**: 문진 과정에서 응급 증상을 감지하면 즉시 응급 안내로 전환합니다.

**수용 조건 (Acceptance Criteria)**:

- AC-1: 다음 키워드/증상 감지 시 응급 모드로 전환한다:
  - 흉통/가슴 통증, 호흡곤란, 의식 저하/실신
  - 대량 출혈, 심한 화상, 골절 의심
  - 갑작스러운 편측 마비, 언어 장애 (뇌졸중 의심)
  - 심한 알레르기 반응 (아나필락시스)
- AC-2: 응급 모드 전환 시 문진을 즉시 중단한다.
- AC-3: 119 전화 연결 버튼을 최상단에 표시한다.
- AC-4: 가장 가까운 응급의료기관 정보를 즉시 표시한다.
- AC-5: 응급 상황 대처 가이드(증상별 응급처치 요령)를 간략히 제공한다.

### 4.2 에이전트 상세 명세

#### 4.2.1 Orchestrator Agent (Supervisor)

- **역할**: 전체 대화 흐름 제어, 에이전트 라우팅, 상태 관리
- **구현**: LangGraph v2 `langgraph-supervisor`
- **상태 관리**: 대화 컨텍스트, 문진 진행 상태, 추천 결과를 Graph State로 관리
- **라우팅 로직**:

```python
# 의사 코드
if user_input == "시작" or state == "INIT":
    route_to("문진 Agent")
elif state == "문진 완료":
    route_to("진단 Agent")
elif state == "진료과 추천 완료" and user_input contains location:
    route_to("검색 Agent")
elif state == "병원 리스트 제시" and user_input == hospital_selection:
    route_to("정보 제공 Agent")
elif emergency_detected(user_input):
    route_to("응급 처리 모듈")
```

#### 4.2.2 문진 Agent

- **역할**: 표준 문진표 기반 대화형 증상 수집
- **입력**: 사용자 자연어 메시지
- **출력**: 구조화된 문진 데이터 (JSON)
- **참조 리소스**: 문진표 Knowledge Base (벡터 스토어)
- **핵심 로직**:
  - 사용자 자유 입력 → NLU로 증상 엔티티 추출
  - 추출된 증상 기반으로 문진표 KB에서 후속 질문 검색
  - 필수 항목 미수집 시 추가 질문 생성
  - 응급 키워드 실시간 모니터링

#### 4.2.3 진단 Agent

- **역할**: 문진 데이터 분석 및 진료과 추천
- **입력**: 구조화된 문진 데이터
- **출력**: 추천 진료과 (1순위, 2순위), 근거, 긴급도
- **참조 리소스**: 증상-진료과 매핑 데이터베이스
- **핵심 로직**:
  - 문진 데이터에서 증상 키워드 추출
  - 매핑 DB 조회 + LLM 기반 종합 판단
  - 복합 증상 시 가중치 기반 우선순위 결정
  - 의료 면책 문구 자동 첨부

#### 4.2.4 검색 Agent

- **역할**: 위치 기반 병의원 검색 및 필터링
- **입력**: 사용자 위치 (좌표 또는 주소), 추천 진료과
- **출력**: 주변 병의원 3개 리스트 (이름, 주소, 거리, 평점, 운영 상태)
- **사용 API**: 네이버 지도 API, 건강보험심사평가원 API
- **핵심 로직**:
  - 주소 → 좌표 변환 (Geocoding)
  - 반경 내 해당 진료과 병의원 검색
  - 거리·평점·운영 상태 가중치 정렬
  - 상위 3개 선별

#### 4.2.5 정보 제공 Agent

- **역할**: 선택된 병원의 상세 정보 조회 및 구성
- **입력**: 병원 식별자 (ID)
- **출력**: 병원 기본 정보, 지도 데이터, 웹페이지 URL
- **사용 API**: 심평원 API, 네이버 지도 API
- **핵심 로직**:
  - 병원 기본 정보 조회 (주소, 전화, 진료시간)
  - 네이버 지도 임베드 데이터 생성
  - 병원 웹페이지 URL 조회
  - 통합 상세 정보 카드 구성

from typing import TypedDict, Optional
from langgraph.graph import StateGraph

class HealthConsultationState(TypedDict):

## 5. Knowledge Base 설계

### 5.1 문진표 Knowledge Base

표준 문진표 데이터를 벡터 스토어에 저장하여 문진 에이전트가 참조합니다.

**데이터 소스**:

- 국가건강검진 일반 문진표
- 진료과별 초진 문진표 (내과, 안과, 정형외과, 피부과, 치과, 이비인후과 등)
- 증상별 추가 문진 가이드

**문서 구조 (청크 단위)**:

```json
{
  "doc_id": "screening_general_001",
  "category": "일반 문진",
  "question": "현재 앓고 있는 질환이 있으십니까?",
  "options": ["고혈압", "당뇨병", "이상지질혈증", "심장질환", "뇌졸중", "기타", "없음"],
  "purpose": "기저질환 파악",
  "follow_up_conditions": {
    "고혈압": ["현재 혈압약을 복용하고 계신가요?"],
    "당뇨병": ["인슐린 또는 경구 혈당강하제를 복용하고 계신가요?"]
  },
  "related_departments": ["내과"]
}
```

### 5.2 증상-진료과 매핑 DB

증상 키워드와 진료과 간의 매핑 데이터를 관계형 DB에 저장합니다.

**테이블 설계**:

```sql
-- 증상 테이블
CREATE TABLE symptoms (
    id SERIAL PRIMARY KEY,
    name VARCHAR(100) NOT NULL,          -- 증상명 (예: "두통")
    category VARCHAR(50) NOT NULL,        -- 카테고리 (예: "신경계")
    body_part VARCHAR(50),               -- 관련 신체 부위
    keywords TEXT[],                     -- 동의어/관련 키워드
    emergency_flag BOOLEAN DEFAULT FALSE  -- 응급 증상 여부
);

-- 진료과 테이블
CREATE TABLE departments (
    id SERIAL PRIMARY KEY,
    name VARCHAR(50) NOT NULL,           -- 진료과명 (예: "신경과")
    description TEXT,                    -- 설명
    common_conditions TEXT[]             -- 주요 진료 질환
);

-- 증상-진료과 매핑 테이블
CREATE TABLE symptom_department_mapping (
    id SERIAL PRIMARY KEY,
    symptom_id INT REFERENCES symptoms(id),
    department_id INT REFERENCES departments(id),
    priority INT DEFAULT 1,              -- 1순위/2순위
    confidence FLOAT DEFAULT 0.8,        -- 매핑 신뢰도
    conditions TEXT                       -- 매핑 조건 (예: "3일 이상 지속 시")
);
```

### 8.1 마일스톤

| 마일스톤 | 기간 | 산출물 |
|---------|------|--------|
| M1: 설계 완료 | 2주 | PRD 확정, 기술 설계서, DB 스키마, API 명세 |
| M2: KB 구축 | 2주 | 문진표 KB, 증상-진료과 매핑 DB, 테스트 데이터 |
| M3: 에이전트 개발 | 3주 | 4개 에이전트 구현 및 단위 테스트 |
| M4: API 연동 | 2주 | 네이버 지도, 심평원 API 연동 및 통합 테스트 |
| M5: 프론트엔드 | 2주 | UI 구현, 에이전트 연동, E2E 테스트 |
| M6: QA 및 런칭 | 1주 | 사용성 테스트, 버그 수정, 배포 |
| **총 개발 기간** | **12주** | |

### 8.2 MVP 범위 (v1.0)

MVP에 포함되는 기능:

- FR-001: 서비스 진입 및 이용 동의
- FR-002: 표준 문진표 기반 AI 건강상담
- FR-003: 증상-진료과 매핑 및 추천
- FR-004: 위치 기반 병의원 검색 (3개)
- FR-005: 병원 상세 정보 표시
- FR-006: 네이버 지도 연동
- FR-008: 응급 상황 감지 및 안내

### 8.3 Post-MVP 로드맵

| 버전 | 시기 | 주요 기능 |
|------|------|----------|
| v1.1 | MVP+4주 | 병원 웹페이지 연동, 상담 결과 요약/저장 |
| v1.2 | MVP+8주 | 상담 결과 공유(카카오톡/PDF), 상담 이력 관리 |
| v2.0 | MVP+16주 | 음성 입력, 병원 예약 연동, 다국어 지원 |

## 9. 리스크 및 완화 방안

| 리스크 | 영향도 | 발생 가능성 | 완화 방안 |
|-------|-------|------------|----------|
| AI 진료과 추천 오류 | 높음 | 중간 | 의료 전문가 검증, 면책 고지 강화, 사용자 피드백 루프 |
| 의료법 저촉 우려 | 높음 | 낮음 | 법률 자문, "의료 행위 아님" 명시, 진단·처방 기능 배제 |
| 네이버 지도 API 비용 증가 | 중간 | 중간 | API 호출 최적화, 캐싱 전략, 비용 임계치 알림 |
| LLM 환각(Hallucination) | 높음 | 중간 | 매핑 DB 기반 검증, 구조화된 출력 강제, 근거 명시 의무화 |
| 사용자 이탈 (문진 길이) | 중간 | 높음 | 문진 3분 이내 완료 목표, 필수/선택 항목 분리, 프로그레스바 |
| 병원 정보 최신성 | 중간 | 중간 | 공공데이터 주기적 동기화, 사용자 제보 기능 (향후) |

## 10. 부록

### 부록 A: 용어 정의

| 용어 | 정의 |
|------|------|
| 문진 (問診) | 환자의 증상, 병력 등을 질문을 통해 파악하는 의학적 과정 |
| 진료과 | 의료기관에서 전문적으로 다루는 의학 분야 (내과, 안과, 정형외과 등) |
| KB (Knowledge Base) | 에이전트가 참조하는 구조화된 지식 데이터베이스 |
| Supervisor 패턴 | 중앙 오케스트레이터가 하위 에이전트를 조율하는 멀티에이전트 아키텍처 |
| PII | 개인 식별 정보 (Personally Identifiable Information) |
| RAG | 검색 증강 생성 (Retrieval-Augmented Generation) |
| NLU | 자연어 이해 (Natural Language Understanding) |
